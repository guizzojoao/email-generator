# Email Signature Generator — Redesign

**Date:** 2026-02-27
**Status:** Approved, pending implementation

---

## Problem

The current generator is a collection of standalone HTML files with signature templates hardcoded directly in JavaScript. Every time a new trade show signature is needed, someone must manually edit the HTML, copy and distribute the file, and update template references by hand. There is no central URL for colleagues to bookmark, no connection between design assets and the generator, and no path for a designer to publish a new signature without engineering involvement.

---

## Goals

- One stable URL all colleagues bookmark — always up to date
- Designer adds a banner image to a folder — new signature appears in the generator automatically, no code changes
- Private repository, publicly accessible site and assets (works in all email clients worldwide)
- Preview tool so designers can verify how a signature looks before publishing

---

## Non-Goals

- Programmatic banner generation (banners are still designed in Photoshop/AE)
- User accounts or authentication for the employee-facing generator
- Anything beyond static file hosting

---

## Architecture

### Hosting

GitHub Pages, deployed from the `main` branch. The repository is private; the Pages site is publicly accessible. All assets served through GitHub Pages URLs are reachable by any email client worldwide.

### Repository structure

```
email-generator/
├── index.html                   # Employee-facing generator
├── preview.html                 # Designer preview tool
├── templates/
│   ├── manifest.json            # Auto-generated by CI — never edited by hand
│   ├── signature-default.html   # Static base template (no banner)
│   └── signature-<slug>.html    # Auto-generated from banners — never edited by hand
├── assets/
│   ├── logo-email.gif           # Company logo
│   └── banner-<slug>.<ext>      # Trade show banners (designer uploads here)
├── scripts/
│   └── generate.js              # Scans assets/, writes templates/ and manifest.json
└── .github/
    └── workflows/
        └── generate-templates.yml
```

### Naming convention

Banner filenames drive everything downstream:

| File in `assets/`        | Generated template                   | Dropdown label |
|--------------------------|--------------------------------------|----------------|
| `banner-cedia-2025.gif`  | `templates/signature-cedia-2025.html`| CEDIA 2025     |
| `banner-ise-2026.gif`    | `templates/signature-ise-2026.html`  | ISE 2026       |
| `banner-hpbexpo-2026.png`| `templates/signature-hpbexpo-2026.html` | HPBEXPO 2026 |

Slug parsing: split on `-`, uppercase alphabetic parts, keep numeric parts as-is, join with spaces.

The `signature-default.html` template is static (no banner). It is always listed first in the dropdown regardless of alphabetical order.

## Components

### `index.html` — Employee generator

The existing `test/main.html` promoted to the root. One functional change: replace the hardcoded `signatureTemplates` array with a fetch of `templates/manifest.json` on load.

```
DOMContentLoaded
  → fetch('templates/manifest.json')
  → populate <select> dropdown
  → restore last-used template from localStorage (or default to first)
  → load template and render preview
```

All other behavior (live preview, copy to clipboard, dark mode, optional phone/meeting fields) is preserved unchanged.

Fallback: if the manifest fetch fails, display a clear error message rather than a broken blank page.

### `preview.html` — Designer preview tool

A standalone page linked from a small "Designer tools" note in the generator footer. No server interaction. Entirely client-side.

**Fields:**
- **Banner image** — file picker, image loaded via `URL.createObjectURL()` for local preview
- **Event name** — auto-populated from the filename (`banner-ise-2026.gif` → `ISE 2026`), editable

**Output:**
- Live preview of the full signature using sample data (Jane Smith, Senior Engineer, sample phone and meeting link) with the selected banner rendered at full width
- A confirmation note showing the exact filename to use: `banner-ise-2026.gif`
- A direct link to the GitHub upload page for `assets/` so the designer can upload in one click

No download buttons. The preview page is a sanity check, not a publishing step — publishing is done by uploading to GitHub.

### `scripts/generate.js`

Plain Node.js, no external dependencies.

**Inputs:** `assets/` directory
**Outputs:** `templates/signature-<slug>.html` for each banner, `templates/manifest.json`

Algorithm:
1. Scan `assets/` for files matching `banner-*.{gif,png,jpg,jpeg}`
2. For each banner file:
   - Parse slug: strip `banner-` prefix and extension
   - Derive display name from slug
   - Generate `templates/signature-<slug>.html` (base template + banner row)
3. Build manifest array — `default` always first, rest sorted alphabetically by display name
4. Write `templates/manifest.json`

Generated template structure (the only difference from default is the banner `<tr>`):

```html
<table ...>
  <tr><!-- logo + name/title/phone/meeting/website --></tr>
  <tr>
    <td colspan="2">
      <img src="../assets/banner-ise-2026.gif" alt="ISE 2026" width="440" />
    </td>
  </tr>
  <tr><!-- legal disclaimer --></tr>
</table>
```

Image paths are relative (`../assets/`), which works correctly on GitHub Pages and when running locally via a dev server.

### `.github/workflows/generate-templates.yml`

Triggers on push to `main` when any `assets/banner-*` file changes.

Steps:
1. Checkout repo
2. Setup Node.js
3. Run `node scripts/generate.js`
4. Commit changed files in `templates/` back to main with message `chore: auto-generate templates [skip ci]`
5. Push

The `[skip ci]` tag prevents the workflow from retriggering on its own commit. GitHub Pages redeploys automatically when the commit lands on `main`.

## Data flow

```
Designer uploads banner-ise-2026.gif to assets/
        ↓
GitHub Actions: generate-templates.yml triggers
        ↓
scripts/generate.js runs:
  - writes templates/signature-ise-2026.html
  - rewrites templates/manifest.json
        ↓
Auto-commit pushed to main [skip ci]
        ↓
GitHub Pages redeploys (~1–2 min)
        ↓
index.html fetches updated manifest.json
  → "ISE 2026" appears in dropdown
```

## End-to-end workflows

### Employee generating a signature

1. Open `<org>.github.io/email-generator`
2. Select template from dropdown (e.g., "ISE 2026")
3. Fill in name, title, and optionally phone and meeting link
4. Click **Copy Signature**
5. Paste into Gmail or Outlook signature settings

### Designer publishing a new show signature

1. Export banner from Photoshop/AE — name it `banner-<show>-<year>.<ext>`
2. *(Optional)* Open `preview.html`, pick the file, verify the full signature looks correct
3. Go to github.com → `assets/` folder → upload the banner file
4. GitHub Action runs (~30 sec), generates template and updates manifest
5. GitHub Pages redeploys (~1–2 min)
6. New signature is live in the dropdown — no further action needed

### Updating an existing banner

1. Export new version of the banner with the same filename
2. Upload to `assets/` on GitHub — overwrites the old file
3. Action regenerates the template automatically
